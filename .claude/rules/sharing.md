# Public Share Rules

Public-share surfaces. Server-only writes via Admin SDK, token IS the access credential, soft-revoke via `revokedAt`, noindex/nofollow on the public page.

- **All share writes go through helpers in `lib/firebase/repos.ts`.** Examples: `mintOrGetShareToken`, `revokeShare`, `getSharedReport`, `listUserShares`, `incrementShareViewCount`. Don't reach into share collections directly — add a typed helper if missing.
- **The token IS the access credential.** 12 base62 chars = 71 bits of entropy. Server-side mint only via `generateShareToken()` in `lib/share/token.ts`. `isValidShareToken()` validates shape before any Firestore read; invalid → `notFound()`.
- **Share collections deny ALL client SDK operations** in `firestore.rules` (`allow read, write: if false`). Public share page + OG image route both read via Admin SDK in `getSharedReport`, which applies the `revokedAt`-null gate server-side. Defense-in-depth against leaked-token uid harvesting.
- **Snapshot is frozen at mint time.** Re-running source synthesis (prompt-version bump, manual cron run) mutates the source doc but does NOT mutate `sharedReports/{token}.snapshot`. Shared URL keeps showing what user actually shared. Mirrors snapshot-at-save patterns.
- **Idempotent on `(uid, key)` (e.g. `(uid, reportDate)`).** Re-clicking share returns existing token, not new one. **Revoke-aware:** if existing row has `revokedAt !== null`, fresh token is minted and revoked row stays as tombstone (audit trail). Old URL keeps returning 404.
- **Revoke is soft delete.** `revokedAt` stamped; doc stays in place. Public `get` rule blocks reads when `revokedAt !== null`. Future cron can hard-delete tombstones older than N days if storage cost matters.
- **View counts are best-effort.** Increment is fire-and-forget from public page render; failure is swallowed. Not load-bearing for any access decision.
- **`/share/d/[token]` is noindex/nofollow.** `metadata.robots: { index: false, follow: false }`. Don't want share pages competing in search; don't want shared content influencing own SEO. OG image at `/share/d/[token]/opengraph-image.tsx` IS reachable by social-platform crawlers (X/LinkedIn/Reddit ignore noindex for OG fetches — that's the point of sharing).
- **Mint route is rate-limited.** `checkRateLimit({ uid, bucket: "share-mint", limit: 10 })`. Re-clicking share is idempotent at data layer (same token returned) but still counts against rate limit — burst of dozens of clicks gets denied with 429.
- **`canShare: true` for every tier post-launch.** Entitlement field exists in `lib/billing/entitlements.ts` so future gating is a one-line change. Adding paid-only share variant later = flip free's `canShare` to `false` and short-circuit the mint route.
- **Adding a new share surface = pass key into `<ShareButton ... />`.** All sources of truth (date inference, repo lookup) live behind that prop. Don't hand-build a `fetch("/api/share/...")` somewhere else — every share entry point through the component so analytics stays consistent.
- **Place share button INSIDE the header of the content it shares, not as floating page-level affordance.** Refactor moved the button into the host section's own header via a `shareSlot?: ReactNode` prop — previously the button sat in a standalone `justify-end` div ABOVE the card, which left users wondering what they were sharing.
- **Copy-link must give explicit confirmation.** Trigger flips label to "Copied ✓" (accent-green) for 1.5s after successful clipboard write, auto-closes menu, announces via `aria-live="polite"`. `navigator.clipboard.writeText` wrapped in try/catch — Permissions Policy / non-secure-context browsers reject it; surface failure inline rather than silently leaving user wondering. Popover dismisses on click-outside and Escape.
- **Share trigger disables itself while a mint is in flight** (`disabled={loading}` + `disabled:opacity-50 disabled:cursor-not-allowed`). Two rapid clicks would otherwise fire two parallel `POST /api/share/*` requests — server-side mint transaction makes result idempotent (same token returned), but second request still burns Firestore round-trip + counts against 10/min `share-mint` rate limit.
- **Analytics event** — `share_clicked` curated in `lib/analytics/track.ts`. Shape per surface (daily-share vs trends-share). Kept PII-free: no uid, no token, no quote text, no filter `value`.
- **Share collection is server-only on every operation.** Even on public read path, Server Component fetches via Admin SDK — no client-side Firestore read of share docs. Keeps rules-engine workload trivial and lets the page do best-effort `viewCount` increments inside a server transaction.
- **For Trends-dashboard-style share docs that include a large array (e.g. scatter points), CAP the array** in the snapshot. Firestore 1MB per-doc limit. Sort newest-first and slice tail; surface `truncated: true` in the snapshot so UI can warn the sharer. Bumping the cap requires reasoning about worst-case doc size — re-derive before changing.
- **Public share viewers don't have Firestore access on owner's source data.** Trim share-doc fields that would enable doc-id harvesting (e.g. `analysisId` on scatter points). Chart click → `/analyses/[id]` is disabled on public surface via `<HistoryChart disableNavigation />` prop.
