# Cloud Functions v2 Rules

Pipeline runs as a single Gen 2 Firestore-triggered function. It is *thin* — almost all logic lives in the parent `lib/pipeline/`, bundled in by esbuild. The function is just the trigger glue.

- **Function entry is `functions/src/index.ts`.** Exports an `onDocumentCreated` trigger on `users/{userId}/analyses/{analysisId}`. Reads `data.status`; if not `"queued"` returns immediately (cache copies and re-triggers).
- **Use `firebase-functions/v2` only.** Gen 1 imports (`firebase-functions/v1`) are deprecated for this project.
- **Region and resources via `setGlobalOptions`.** Currently `us-central1`, `maxInstances: 10`. Override per-function with the options bag (e.g. `timeoutSeconds: 540, memory: "1GiB"` for long-running pipeline).
- **Secrets, not env vars.** Declare via `secrets: ["GEMINI_API_KEY"]` on the function definition. Set once with `firebase functions:secrets:set`. Don't put LLM keys in runtime env config. App Hosting uses the same Secret Manager backend.
- **Sentry release pinned to `K_REVISION`.** Cloud Functions Gen 2 + App Hosting both run on Cloud Run; `K_REVISION` is auto-injected to the current revision id. Every captured event tagged to a specific deploy. `SENTRY_RELEASE` (explicit override) wins when set.
- **Caught errors MUST NOT throw out of the handler.** `runAnalysis` writes `status: "failed"` itself. Throwing would race with Cloud Functions Gen 2 retry semantics — could leave doc in weird state, double-bill, or double-process. Function entry should also `return` on no-op cases (non-`queued` status), not throw.
- **Bundling via esbuild** (`functions/build.mjs`). Single ESM file at `functions/lib/index.js` with `firebase-admin` and `firebase-functions` left external (resolved by Functions runtime). Pulls in `../../lib/pipeline`, `../../lib/llm`, etc.
- **Build before deploy.** `firebase deploy --only functions` runs the predeploy hook (`pnpm --dir functions install --frozen-lockfile=false && pnpm --dir functions run build`). Confirm in `firebase.json`. Don't deploy without rebuilding; stale `lib/index.js` can ship to prod.
- **Local dev typically doesn't need the Functions emulator.** `/api/analyze` route detects `NODE_ENV !== "production" && !FUNCTIONS_EMULATOR` and runs `runAnalysis()` in-process so `pnpm dev` works alone. Start emulator only to test trigger wiring.
- **`functions/lib/` and `functions/node_modules/` are build artifacts.** Gitignored, ESLint-ignored. Do not edit `functions/lib/index.js` by hand.
- **Don't import client-side Firebase from `functions/`.** Web SDK won't run in Functions runtime; use `firebase-admin` only.
- **Audio fallback runs server-side.** When captions fail, pipeline calls `transcribeYouTubeUrlWithGemini({ youtubeVideoId })`. Gemini fetches the watch URL on Google's backend (sidesteps yt-dlp / bot-detection from Cloud Run IPs).
- **Hourly crons (`0 * * * *`) use per-user local-time gating, backed by precomputed `localHourBucket` field.** Indexed `usersCol.where("localHourBucket", "in", bucketHoursForUtc(currentUtcHour))` over ±1-hour window (DST safety). Per-user `localHour(user.timezone, now) === N` filter on each match for defense-in-depth against stale buckets. Reads scale O(users firing this hour), not O(all users). DST fall-back guard in synthesis (`getDailyReportByDate` check) prevents double-billing on autumn DST.
- **Per-user fan-out batched via `Promise.allSettled` in groups of 10.** Wall-time drops from O(N × per-user-latency) to O(N/10 × max-batch-latency). Counter increments inside async callbacks safe without a mutex — Node.js single-threaded, each `await` is the only interleave point.
- **Outer try/catch/finally with Sentry on every cron handler.** Inner per-user try/catch covers per-user errors; outer catches throws from the BUCKET QUERY itself OR cycle-completion log. Outer catch fires `captureException(err, { site: "cronName.outer" })` then re-throws so GCF still logs the crash. Without outer catch, a Firestore blip silently produces zero per-user fires for the entire cycle with no Sentry signal.
- **Per-doc trigger handlers also need outer catch with stuck-queued recovery.** `analyzeVideoOnCreate` wraps body in try/catch — `runAnalysis` catches and writes `failed` per no-throw contract, but the OTHER throw paths (user-doc read before runAnalysis) are uncovered. Without outer catch, a Firestore blip during that read crashes the function and leaves analysis stuck at `status: "queued"` forever (GCF Gen 2 doesn't auto-retry document triggers). The catch fires Sentry capture + makes a best-effort `setAnalysisStatus({ status: "failed" })` write so the doc moves out of queued.
